Write a technical blog post in Markdown based on the following author instructions:

Title hint: {{TITLE_HINT}}

Instructions:
{{INSTRUCTIONS}}

Requirements:
  - Begin with a machine-readable TAGS line listing 3–6 comma-separated, lowercase, hyphenated tags relevant to the post (e.g. TAGS: haskell, functional-programming, type-system)
  - You may refine the title from the hint to make it specific and compelling; the hint is just an idea, not a constraint
  - Follow the TAGS line with the bilingual content using the exact format described below
  - Begin the body immediately after the title — no greetings, no preamble, no "In this post we will...". Open with the first substantive sentence of the topic itself
  - Include multiple practical code examples with real-world context if applicable; annotate non-obvious idioms
  - Aim for depth: cover edge cases, common pitfalls, and engineering trade-offs
  - Length: 1500–2000 words per language version (excluding the TAGS line and separators)
  - End each version with a '## Further Reading' section listing relevant online resources as bullet points using the format `- [Resource title](url)` — include real, stable URLs only
  - The Portuguese version must be a complete, natural-language translation of the English version (not a machine word-for-word translation); idiomatic Brazilian Portuguese throughout
  - Markdown formatting rules (strictly enforced):
    - The post title must be an H1 (`# Title`) — never start with H2 or lower
    - Every heading marker must be followed by exactly one space before the heading text (e.g. `## Section`, never `##Section`)
    - Blank lines are required before and after every heading, code block, and list
    - Fenced code blocks must always specify a language identifier (e.g. ```haskell, never ``` alone)
    - Inline code uses single backticks; never use bold or italics as a substitute for code formatting

Output ONLY the following structure (no extra text before or after):
